### 1-2. 深度レベル定義

#### Lv1: 情報提供（5%）
**特徴:**
- 単なる情報共有
- 自分の意見・解釈は最小限
- 読者への価値は「知る」こと

**例:**
```
情報処理技術者試験、27年度から大幅刷新。
日経からも発表ですね。
```

---

#### Lv2: 戦略的提言（35%）
**特徴:**
- 「何をすべきか」の方向性を示す
- 一般論と自分の経験を組み合わせる
- 読者への価値は「意思決定の指針」

**例:**
```
資格取得が無駄かどうかみたいな議論があるけど、
無駄だと感じる人は実務に結びつかない資格を選択しているか、
結びつける意識が低いと感じる。

実務と資格は相乗効果。
実務で学んだことを資格勉強で整理できるし、
資格でインプットしたことを実務でアウトプットもできる。
選択と意識次第。
```

**深度:**
- 議論の対立軸を認識
- 一般論への反論
- 具体的な「相乗効果」の提示
- 行動指針（「選択と意識次第」）

---

#### Lv3: 実践的ナレッジ（45%）
**特徴:**
- 「どうやるか」の具体論
- 失敗体験→原因→対策の構造
- 読者への価値は「明日から使える」

**例1（責務分離）:**
```
AIで実装してる人は責務分離の考え方できてますか？
責務分離とは、
「関心ごとを分けること」を意味し、
ソフトウェアの機能や役割を切り分けて、
それぞれを独立して管理できるようにする考え方です。

ただし！
・クラスを増やすこと
・ファイルを細かく分けること
ではない。
```

**深度:**
- 概念の定義（「関心ごとを分ける」）
- 誤解の指摘（「ただし！これではない」）
- 「何をやるべきでないか」の明示

**例2（設計レビュー）:**
```
設計レビューでよく詰まる原因。

「何を作るか」説明できるけど、
「なぜその構成なのか」説明できない。

対策はシンプルで、
設計書を書く前に必ずこれを言語化👇

・この設計で「何を守りたいか」
・他案を捨てた理由
・将来変更されそうなポイント

これが書けると、レビューの質が一気に変わる。
```

**深度:**
- 具体的な失敗パターン（「何を作るか」は言えるが「なぜ」は言えない）
- 失敗の本質的抽出
- 具体的な解決策（3つの言語化項目）
- 結果の保証（「レビューの質が一気に変わる」）

---

#### Lv4: 価値観転換（15%）
**特徴:**
- 常識・前提を問い直す
- パラダイムシフトを提示
- 読者への価値は「思考の枠組みそのものの変革」

**例:**
```
AIがコードを書く時代に、
エンジニアはどうやって稼ぐのか。

答えは、
「実装」ではなく
「設計・判断・説明」ができること。
```

**深度:**
- 社会の変化（「AIがコードを書く時代」）の認識
- 根本的な問い（「どうやって稼ぐのか」）
- パラダイムシフト（「実装」→「設計・判断・説明」）
- 新しい価値基準の提示

**別例:**
```
Xだと怖いほど思考が似ている人が多くて嬉しい。
AIで自動化しまくってますよって言ってる外資系企業のエンジニアは、
書けるけど書かないだけのスーパーエンジニアですよ。
経営者含めて。
```

**深度:**
- 表層的な見方への批判（「AIで自動化しまくってます」と言ってる人）
- 本質の洞察（「書けるけど書かないだけ」）
- 隠れた事実の暴露（「経営者含めて」）
- 価値観の転換（AIに書かせること≠実装力がない）

## 3. 高品質ツイートの構造分析

### 3-1. Lv3 実践的ナレッジの標準フォーマット

```
[問いかけ：実践上の課題]
↓
[課題の具体化：何が起きているか]
↓
[原因の深掘り：なぜ起きるのか]
↓
[「ただし！」で誤解を訂正]
↓
[具体策：箇条書き]
↓
[結果の保証：こう変わる]
```

**実例（設計レビュー）:**
```
設計レビューでよく詰まる原因。

「何を作るか」説明できるけど、
「なぜその構成なのか」説明できない。

対策はシンプルで、
設計書を書く前に必ずこれを言語化👇

・この設計で「何を守りたいか」
・他案を捨てた理由
・将来変更されそうなポイント

これが書けると、レビューの質が一気に変わる。
```

**構造分解:**
1. 問い: 「設計レビューでよく詰まる原因」
2. 現象: 「何を作るか」は言えるが「なぜ」は言えない
3. 対策: 3つの言語化項目
4. 結果: 「レビューの質が一気に変わる」

---

### 3-2. Lv4 価値観転換の標準フォーマット

```
[時代背景の提示]
↓
[根本的な問い]
↓
[答え：パラダイムシフト]
↓
[具体例：新しい価値観の適用範囲]
```

**実例（AI時代の稼ぎ方）:**
```
AIがコードを書く時代に、
エンジニアはどうやって稼ぐのか。

答えは、
「実装」ではなく

```

**構造分解:**
1. 背景: 「AIがコードを書く時代」
2. 問い: 「どうやって稼ぐのか」
3. 答え: 「実装」→「設計・判断・説明」への転換
4. 具体化: このアカウントで提供する価値

## 4. 価値あるエンジニアナレッジの定義

### 4-1. 高価値ナレッジの条件

1. **具体性**
   - 抽象論で終わらない
   - 「どうやるか」まで示す
   - 失敗体験に基づく

2. **反直觉性**
   - 「あれ？そうなんだ」と思わせる
   - 常識を疑う
   - 誤解を訂正する

3. **実践性**
   - 明日から使える
   - 手順が明確
   - 結果が保証される

4. **本質性**
   - 現象ではなく構造を捉える
   - 一度の解決で多次元の問題が片付く
   - 再利用可能な原則

### 4-2. 低価値ナレッジのパターン

| パターン | 例 | なぜ低価値か |
|---------|---|-------------|
| 情報の転載 | 「試験が変わります」 | 自分の考えがない |
| 一般論 | 「継続が大事」 | だれでも言える |
| 当たり前の結論 | 「勉強しましょう」 | 新規性がない |
| 感想だけ | 「嬉しいです」 | 読者の価値にならない |

## 5. トピック選定の思考プロセス

### 5-1. 高品質トピックの選び方

**ステップ1: 自分の失敗から問いを立てる**
```
失敗: 設計レビューで詰まった
→ 問い: なぜ詰まるのか？
→ 原因: 「なぜその構成なのか」言えない
→ 対策: 設計前に言語化する
```

**ステップ2: 世の中の議論に反証を立てる**
```
議論: 「資格は無駄」
→ 反証: 意識次第で相乗効果がある
→ 具体化: 実務×資格の相互補完
→ 結論: 選択と意識次第
```

**ステップ3: 常識的前提を疑う**
```
前提: エンジニアはコードを書くもの
→ 疑い: AI時代はどうなる？
→ 転換: 「実装」→「設計・判断・説明」
→ 具体化: このアカウントで何を提供するか
```

### 5-2. トピックの深度を深める方法

**Lv1→Lv2に深める:**
```
Lv1: 試験が変わります
↓
Lv2: どう対応すべきか？
→ 新範囲の学習
→ 旧範囲の整理
→ 移行期間の戦略
```

**Lv2→Lv3に深める:**
```
Lv2: 資格は勉強しましょう
↓
Lv3: 具体的にどう勉強するか？
→ 実務での学びを資格勉強で整理
→ 資格で学んだことを実務でアウトプット
→ 相乗効果を最大化する学习方法
```

**Lv3→Lv4に深める:**
```
Lv3: 設計を勉強しましょう
↓
Lv4: なぜ設計が必要なのか？
→ AIがコードを書く時代
→ 「実装」の価値が下がる
→ 「設計・判断・説明」の価値が上がる
```

## 6. 生成時のトピック・深度ガイド

### 6-1. 推奨トピック割合（1週間分）

| 深度 | 割合 | 週あたり件数 |
|------|------|-------------|
| Lv4 価値観転換 | 15% | 1件 |
| Lv3 実践的ナレッジ | 45% | 3件 |
| Lv2 戦略的提言 | 30% | 2件 |
| Lv1 情報提供 | 10% | 1件（稀） |

### 6-2. 深度別の生成ガイド

**Lv4 価値観転換:**
```
[時代背景] + [根本的問い] + [パラダイムシフト] + [具体例]

例:
AIで自動化してる人、
バイブコーディング卒業できてますか？

答えは、
「コードを書く」ではなく
「AIのコードをレビューする」こと。

この違い、
・AIが出した答えを検証する力
・仕様の意図を読み取る力
・変更時の影響範囲を即座に理解する力
```

**Lv3 実践的ナレッジ:**
```
[実践上の課題] + [現象の具体化] + [「ただし！」誤解訂正] + [具体策]

例:
Claude Code使ってますか？
便利だけど、サブエージェントの使い方が分かってない人が多い。

ただし！
・並列実行すればいいわけじゃない
・エージェント種類を増やせばいいわけではない
独立したコンテキストでレビューさせること。
これがバイアスを排除して、本当の第三者視点を出す鍵。
```

**Lv2 戦略的提言:**
```
[世の中の議論] + [自分の意見] + [理由/具体例] + [結論]

例:
AIは人間を replace するって言われてるけど、
俺は「補完」すると思ってる。

AIは設計ができない
AIは責任を取れない
AIは文脈を全ては理解しきれない

使う場面と疑う場面の判断力が、
これからのエンジニアの差別化ポイント。
```

**Lv2-4 AI規制・ガバナンス系（国家戦略視点）:**
```
[問いかけ] + [現状の具体化] + [「ただし！」誤解訂正] + [バランス論]

例:
AI戦略、慎重になりすぎてませんか？

2025年5月にAI推進法は成立したけど、
EUは2024年8月にもうAI Act施行。
米国は政権交代で非規制・投資へシフト。

・個人情報保護の意識が強く、データ活用が制限される
・リスクばかり強調されて導入が進まない業界がある
・世界のAI競争で優位に立つのが難しくなってる

「規制して止める」から
「使って最適化する」へ。
このバランスが、これからの日本に必要。
```

**注意（AI規制系トピック）:**
- 事実に基づく（2025年5月AI推進法成立、GDPR/AI Act等）
- 少数派過ぎる意見を当たり前のように言わない
- 「日本は完全に遅れている」等の極端な断定は避ける
- 「規制とイノベーションのバランス」という大方の意見に沿う
