# 投稿価値ルール（信頼残高基準）

全ツイート種別で適用。投稿の「価値」を判定する最上位基準。
出典: @shinkaron「毎月平均5000フォロワーづつ増やしてるけど、耳の痛い話をしてやんよ」

## 根幹原則: 信頼残高

フォロワー数は積み上げてきた信頼の残高。各投稿は信頼残高の増減に直結する個々の取引。
この前提を持たずに投稿を量産しても、長期的な成果にはつながらない。

## 価値提供の5条件（1つ以上を満たすこと）

投稿が「読み手の時間を投資してよかった」と思わせるには、以下のいずれかを達成する必要がある:

| 条件 | 具体例 |
|------|--------|
| **思考が一段深まった** | 既知の話題に新しい視点・切り口を提示 |
| **感情を大きく衝き動かされた** | 驚き・共感・危機感など感情の動きがある |
| **行動のきっかけを与えられた** | 「試してみよう」「調べてみよう」と思わせる |
| **パラダイムシフトを起こされた** | 前提を覆す、常識の再定義 |
| **有用な知識やノウハウが提供された** | 具体的な手順・数値・比較・ツール紹介 |

**5条件のいずれも満たさない投稿は信頼残高を減らす。投稿しない方がマシ。**

## 時間責任の原則

「一投稿ごとに、他者の時間と注意力に対して責任を負え」

読み手はフォロワーの有限な時間を使ってその投稿を読んでいる。
時間を浪費させられたと感じさせる投稿は、信頼残高を削る。

## 逆算設計

読者心理から逆算してポストを設計する。

```
❌ 思いつき → 投稿 → 反応を祈る
✅ 読者が得る価値を定義 → その価値を最大化する構成を設計 → 投稿
```

「このツイートを読んだ人は何を得るか？」を投稿前に必ず自問する。

## 人格投影の原則

発信者の人格・知識・能力・経験・実績をいかに投影させるかが決定的に重要。
誰が書いても同じ内容（一般論の要約、ニュースの転載）では信頼残高は増えない。

persona-dbの情報を活用し、以下を投稿に反映する:
- **経験**: 実際に使った・試した・感じたことベースで書く
- **知識**: 表面的な紹介ではなく、理解の深さが伝わる言及をする
- **スタンス**: 中立的な情報整理ではなく、自分の立場を明確にする

## 読者アクション起点の具体性原則

ナレッジシェア系のツイートでは、読者が「自分もやってみよう」「自分も気をつけよう」と思える具体性を維持する。露出制御やプライバシーのために情報を抽象化しても、読者視点でのアクション可能性を殺さない。

### 具体性を維持する軸
- **数値**: コスト・件数・比率など定量的な情報は省略しない
- **手段**: 何を使って・どう指示して達成したかを含める（行動主体の明示）
- **教訓**: 読者が自分の状況に重ねて「注意しよう」と思えるポイントを残す

### 判定例
```
❌「APIコストを最適化した」
  → 抽象的。読者は何も学べない

✅「$2.33/day → $0.27/dayに落ちた」
  → 数値があるので「自分のAPIコストも確認しよう」と思わせる

❌「クエリの重複に気をつけた方がいい」
  → 一般論。読者の行動を促せない

✅「機能を足したときにいきなり料金が跳ね上がるの見落としがち」
  → 読者が自分の状況に重ねて「確認しよう」と思える具体的な教訓
```

**原則: 実際にやっているかどうかは重要ではない。読者に価値あるナレッジを提供できているかが焦点。**

## BIPトピック選定の優先順位

BIP（Build in Public）は「自分の体験日記」ではない。読者への価値提供が最優先であり、自分が経験した内容に限定する必要はない。

### 優先順位
1. **情報の価値**: 読者にとって有用なナレッジか（最重要）
2. **整合性**: persona-dbのスキル・ツール・知識レベルと矛盾しないか
3. **当事者性**: 実際に経験した内容か（あればプラスだが必須ではない）

### なぜ体験縛りが有害か
- zeitgeistやバズツイート分析で得た知見・トレンドを活用できなくなる
- ネタの幅が極端に狭くなり、内輪感が出る
- 読者にとっての価値よりも「嘘をつかない」ことが優先されてしまう

### 判定例
```
❌ 自分が今日やったことだけをBIPにする
  → 「APIコスト最適化した」「スクリプト書いた」ばかりになる

✅ バズ分析やzeitgeistで見つけた有用なナレッジをBIPとして発信
  → persona-dbのスキル・知識と矛盾しない範囲で、読者に価値ある情報を提供

❌ 経験していないことを「やった」と嘘をつく
  → 嘘は禁止。ただし「〜してみた」ではなく「〜という知見がある」のフレーミングで書ける
```

**原則: BIPの価値 = 読者へのナレッジ提供。体験は信頼性のブーストであり、必須条件ではない。**

## バズツイートシード戦略（ネタの種として活用）

蓄積されたバズツイート（`buzz-tweets-latest.json`, `themed-buzz-*.json`, `buzz_content_evaluations.json`）を、ツイート生成の**種（シード）**として積極的に活用する。

### なぜこの戦略が必要か
- 「メタ言及するな、実践を見せろ」というルールに対し、自分の実体験だけではネタが枯渇する
- バズツイートには実際にバズった知見・tips・体験談が含まれており、信憑性の高い素材になる
- 種から加工して独自コンテンツにすることで、品質とオリジナリティを両立できる

### シード選定基準
蓄積データから以下の条件で種ツイートを選ぶ:
1. **content_type**: opinion, bip, how-to, ai_news, data_insight を優先
2. **originality**: 3以上（独自の切り口があるもの）
3. **virality_factor**: relatability, information_value, entertainment を優先
4. **persona整合性**: persona-dbのスキル・ツール・知識レベルと矛盾しないネタ

テーマ特化バズ（`themed-buzz-*.json`）は一般バズより精度が高いため優先的に参照する。

### 加工ルール（同一認識不可能レベルまで変換）
種ツイートをそのまま使うのは禁止。以下の加工を経て**元ツイートとの同一認識がほぼ不可能**なレベルまで変換する:

1. **視点の転換**: 元ツイートの主張を自分のペルソナの文脈・体験に置き換える
2. **具体例の差し替え**: 元ツイートのツール名・数値・事例を、persona-dbに整合する別の具体例に変更
3. **構造の再構成**: 元ツイートの論理構造（問題提起→解決策→感想）を別の構造に組み替える
4. **口調の適用**: expression-rules.mdの口調ルール（中性的〜女性的な柔らかさ）を全面適用
5. **独自洞察の付加**: 種ツイートにない独自の気づき・反転・深掘りを必ず1つ以上追加

### 加工度の判定基準
```
元ツイート（シード）:
  「Claude Codeで並列レビューさせたら3倍速くなったけど、コンテキスト消費がえぐい。損益分岐点が見えない」

❌ 加工不足（同一認識可能）:
  「Claude Codeで並列レビューしたらコンテキスト消費がすごかった。コスパの見極めが難しい」
  → 言い換えただけ。構造も具体例もほぼ同じ

✅ 十分な加工（同一認識ほぼ不可能）:
  「Task toolで7エージェント並列してみたら1セッションでMAXの20%消費しちゃって、結局2エージェントずつ直列にしたら品質も上がった。並列は正義じゃないかもしれない」
  → 具体例差し替え（並列レビュー→Task tool 7エージェント）+ 逆の結論（並列が常に良いとは限らない）+ 独自洞察追加
```

### 禁止事項
- **種ツイートの文面をそのままコピー・微修正して使うこと**
- **種ツイートの著者名・URLを出力に含めること**（あくまで内部参照）
- **persona-dbと矛盾する体験を捏造すること**（「やった」フレーミングはpersona整合時のみ）

**原則: バズツイートは「ネタの方向性」を借りるだけ。表現・構造・具体例・洞察は全て独自に構築する。**

## メタ言及の禁止（概念について語るな、実践そのものを見せろ）

「think in public」「BIP」「CLAUDE.md活用」等の概念・手法について**語る**ツイートは禁止。その概念を**実践している**ツイートを書く。

### なぜメタ言及がダメか
- 「think in publicって大事だよね」→ 概念の説明・推奨であり、think in publicの実践ではない
- 概念を語ると「教える人」ポジションになり、読者は「ふーん」で終わる
- 実践そのものを見せると、読者は思考プロセスを追体験でき、5条件の「思考深化」を満たしやすい

### 判定例
```
-- 「think in public」の場合 --

 メタ言及（概念について語っている）
  「CLAUDE.mdに失敗を書き残すのがthink in publicの良さだよね」
  → 概念を説明・推奨しているだけ。実践ではない

 実践そのもの（思考過程を公開している）
  「プロジェクトのCLAUDE.mdにエラーパターン書き溜めてたら、同じバグを二度踏まなくなった。ただファイルが肥大化してきて、SKILL.mdに分離すべきか悩んでる」
  → 実際の思考・判断・迷いを見せている。これがthink in public

-- 「BIP」の場合 --

 メタ言及
  「Build in Publicはフォロワー獲得に効くと思う」
  → BIPという概念について語っている

 実践そのもの
  「APIコスト$2.33/day→$0.27/dayに落とした。grok-4-fastに変えただけなんだけど、クエリの飛ばし方も変えないとレート制限に引っかかる」
  → 実際の開発過程を見せている。これがBIP
```

**原則: 概念のラベル（think in public, BIP等）を本文に書いた時点でメタ言及の疑い。実践するなら概念名を出す必要はない。**

## 技術トピックのファイル設計ガイダンス

Claude Code / AI開発ワークフローについてツイートする際、具体的なファイル名・設計判断に言及する場合は、**読者が自分のプロジェクトで再現できるレベルの実践的ガイダンス**を含める。

### なぜ必要か
- 「CLAUDE.mdに書く」だけでは、グローバルCLAUDE.md（`~/.claude/CLAUDE.md`）なのかプロジェクトのCLAUDE.md（`./CLAUDE.md`）なのか読者には分からない
- 「エラー対応はどこに書く？」という実務的な疑問に答えないと、読者のアクション起点にならない

### 必須ルール
1. **CLAUDE.mdのスコープは常に明示**: 「グローバルのCLAUDE.md」「プロジェクトのCLAUDE.md」と書き分ける
2. **ファイル設計の選択肢に言及**: CLAUDE.md / SKILL.md / 専用.md（エラーハンドリング.md等）のどれが適切かの判断基準を含められるとベスト
3. **読者が「自分ならどこに書くか」を判断できる情報を残す**

### 判定例
```
 スコープ不明
  「CLAUDE.mdに失敗パターンを書き溜めてる」
  → グローバル？プロジェクト？読者は再現できない

 スコープ明示
  「プロジェクトのCLAUDE.mdにエラーパターンを書き溜めてる」
  → 読者が「自分もプロジェクトのCLAUDE.mdに書こう」と判断できる
```

## 自己テスト（投稿前チェック）

生成したツイートに対して以下を自問する:

1. **5条件チェック**: 5つの価値条件のうち、どれを満たしているか？ → 1つも該当しないなら廃棄
2. **時間責任チェック**: 読み手の時間に見合う価値があるか？ → 「ふーん」で終わるなら不合格
3. **人格投影チェック**: この人が書いた意味があるか？ → 誰でも書ける内容なら不合格
4. **自分読みテスト**: 自分のTLに流れてきたら読みたいか？ → Noなら即廃棄
