# 議論進行度マップ（Discourse Freshness）

## 前提: Claudeの一般論バイアスへの自覚（MANDATORY）

デフォルトのClaudeは**一般論を答える傾向が強い**。
「AIがコード書く時代、設計力が大事」「プロンプトが重要」「学び続けることが大切」等、誰もが既に知っている正論を生成しがち。

この傾向はツイート生成で致命的:
- X上では「それ半年前から言われてるよね」で即スルーされる
- 正しいが退屈 = エンゲージメント0
- 一般論のまとめサイト化 = フォロー価値なし

**本ドキュメントの目的**: 一般論に着地する前に「その論点は議論のどの段階にあるか」を確認し、**最前線の地点から発言する**ための基準を提供する。

## 原則

- ネタの鮮度は「発表日」だけでは測れない
- 同じ話題でも「議論がどこまで進んだか」で鮮度が変わる
- 3ヶ月前に最前線だった論点が、今は消化済みになっていることがある
- 逆に、半年前の話題でも「まだ解決されていない」なら最前線のまま

## 判定フロー

```
1. バズツイートの論点を特定する
2. その論点がこのドキュメントの「進行マップ」のどの段階にあるか確認
3. 判定結果に応じて対応:
   - [消化済] → この論点では書かない。次の段階を探す
   - [主流化中] → そのまま書くとタイミング遅い。主流化の先の問題点を切り口にする
   - [最前線] → ここから書く。ただし「最前線」は数週間で変わるので要注意
   - [未マッピング] → Phase 2のGrok APIで「この論点は今どの段階？」を確認してから判断
```

## 段階の定義

| 段階 | 意味 | X上での反応 |
|------|------|------------|
| **[消化済]** | 大方が認識している。言っても「知ってる」で終わる | スルーされる。エンゲージメント低い |
| **[主流化中]** | 実践者が増えている段階。まだ議論はあるが新規性は薄い | 「あーそれね」感。よほど切り口が独自でないと埋もれる |
| **[最前線]** | まだ答えが出ていない。実践者の間で試行錯誤中 | 共感・議論を呼びやすい。「それ僕も思ってた」が来る |
| **[未到達]** | ほぼ誰も議論していない。先取りしすぎで共感を得にくい | 届かない可能性が高い。タイミングを待つか、啓蒙型なら長文向き |

## ツイート方向別の適用

### BIP方向
- BIPは「自分の体験」が主体なので、進行マップの影響は比較的小さい
- ただし体験の文脈として一般論を使う場合は注意（例: 「設計力が大事だと思った」→ 消化済み論点）
- BIPの鮮度は「体験のリアルタイム性」で担保される

### エンジニアの生き方方向
- **最も進行マップの影響を受ける方向**。論点の鮮度がそのままツイートの鮮度になる
- [消化済]の論点でツイートすると「いつの話してるの」感が出る
- [最前線]の論点を捉えて「問い」を立てるのが最も刺さる

### AIニュース方向
- ニュースの鮮度は `news-freshness-checker` が担当（発表日ベース）
- 進行マップは「ニュースの文脈づけ」に使う（「この発表で議論がどう進むか」の考察部分）

## AIコーディング領域の議論進行マップ

### コードレビュー / 品質担保

| 段階 | 論点 |
|------|------|
| [消化済] | AIが書いたコードは人間がレビューすべき |
| [消化済] | レビューもAIにやらせる（PR Agent等） |
| [主流化中] | agent teamsで並列レビュー（複数AI視点で同時チェック） |
| [最前線] | agent teamsはコスパが悪い → 結局泥臭い手法に回帰しがち → 革新的な解がまだ出ていない |

**X上の温度感**: 精度は大事だがそれを上げる方法の「高速化」にフォーカスが向いている。慎重なエンタープライズ的やり方はウケない。

**現在の前提レイヤー**: 実務者は「マルチエージェントレビュー＋テスト自動化を経て、人間は最終確認のみ」を当たり前として話している。「AIコードは人間がレビューすべき」は結論として正しいが、この前提プロセスを省略して語ると解像度が低く見える。

### Vibe Coding / ドキュメント駆動開発

| 段階 | 論点 |
|------|------|
| [消化済] | CLAUDE.mdに指示を書くと精度が上がる |
| [消化済] | 6定義書 + 3記憶ファイルのフレームワーク（梶谷氏の紹介で再浮上したが元記事は以前から存在） |
| [主流化中] | ドキュメント駆動はデファクト化しつつある |
| [最前線] | コンテキスト量の最適化問題 → 何を書いて何を書かないか。全部書くとコンテキストが溢れる。書かないと精度が落ちる。このバランスの最適解がまだ出ていない |

**現在の前提レイヤー**: 実務者は「CLAUDE.md等のルールファイルでAIに指示を与えるのは当然」として話している。「CLAUDE.mdを書けば精度が上がる」は入門者向けの話であり、今の議論は「何をどこまで書くか」の最適化問題。

### AIコーディングと人間の役割

| 段階 | 論点 |
|------|------|
| [消化済] | AIがコード書くから人間は設計力が大事 |
| [消化済] | プロンプト力が新しいスキル |
| [消化済] | エンジニア不要論（出ては消えるサイクル） |
| [主流化中] | 「AIに任せきりにすると足元掬われる」警鐘（kazunori_279系） |
| [最前線] | 全部一人でできるようになった孤独感 / チーム不要化の心理的インパクト |
| [最前線] | AI文章力が人間超え → コード以外の領域（文章・分析・企画）にも波及し始めている実感 |

**現在の前提レイヤー**: 実務者は「AIでコード・文章・分析が一人で回せる」を前提として話している。「エンジニア不要論」「プロンプト力が大事」は過去の議論であり、今は「全部できるようになった上で、何に集中すべきか」の段階。

### AI文章力 / 創作領域への波及

| 段階 | 論点 |
|------|------|
| [消化済] | AIが文章を書ける |
| [消化済] | AIの文章はまだ人間に劣る |
| [主流化中] | 特定モデル（Opus 4.6等）の日本語力が「普通に上手い」レベルに到達 |
| [最前線] | AI将棋的な「人間が普通に勝てない」現象がコード→文章→クリエイティブ全般に波及しつつある実感 |

**現在の前提レイヤー**: 実務者は「特定モデルの日本語が上手い」を既知として話している。「AIが文章書ける」は当然で、今は「AIの文章力に人間がどう向き合うか」が議論の中心。

### AIエージェント / MCP / Agentic Engineering

| 段階 | 論点 |
|------|------|
| [消化済] | AIエージェントという概念の紹介（AutoGPT等） |
| [消化済] | MCPでツールを接続するとAIが外部操作できる |
| [主流化中] | Claude Code + MCP + Skills で個人がエージェントシステムを構築する実践（vibe coding → agentic engineering への呼称移行） |
| [主流化中] | OpenAI Agents SDK / Google ADK 等のフレームワーク乱立と選定問題 |
| [最前線] | エージェント同士のオーケストレーションコスト問題 → 並列で走らせるとトークン消費が爆発 → 最適構成がまだ見えていない |
| [最前線] | 「全自動」の幻想と実態のギャップ → human-in-the-loopの最適介入ポイントが人によって違う |

**X上の温度感**: 「agentic engineering」というワードが急速に広まりつつあるが、実態として何が「vibe coding」と違うのか明確でない。実践者は試行錯誤中で、華やかなデモと泥臭い実運用の間にギャップがある。

**現在の前提レイヤー**: 実務者は「MCPでツール接続、CLAUDE.mdでルール定義、Task Schedulerで定期実行」を当然の前提として話している。「MCPすごい」「エージェント便利」は入門者の感想であり、今は「オーケストレーションのコスト最適化」「人間がどこで介入すべきか」が議論の中心。

## マップの更新ルール

### 手動更新（セッション中）
- ユーザーから「これ古い」「今はこうだよ」等の鮮度フィードバックを受けた場合、即座にマップを更新する
- Phase 2のGrok API結果で新しい段階を発見した場合も追記する

### 自動更新（Task Scheduler: 毎週日曜 20:00 JST）
- **スキル**: `C:\Users\Tenormusica\.claude\skills\discourse-freshness-updater\SKILL.md`
- **バッチ**: `run_discourse_freshness_updater.bat`（`claude -p` でヘッドレス実行）
- **処理**: Grok APIで英語/日本語の直近議論を調査 → 段階変更を検知 → ファイル更新 → git push → Discord通知
- **コスト**: 約5-7円/回（Grok API 7クエリ）、月約20-30円
- **Discord #en-buzz-tweets**: Bot Token経由で自動取得（`fetch_discord_buzz.js`、バッチ事前実行）

### 更新時の注意
- [消化済]に降格した論点を削除しない（「なぜ古いか」の根拠として残す）
- [最前線]の論点には発見日を付記する（鮮度の目安）
- 新しい領域が出たらセクションを追加する（AIエージェント、マルチモーダル等）
